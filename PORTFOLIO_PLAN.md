# Portfolio Improvement Curriculum

이 문서는 2026년 6월 25일부터 2026년 6월 29일까지 포트폴리오 사이트를 집중적으로 개선하기 위한 실행 계획입니다.

목표는 단순히 화면을 예쁘게 다듬는 것이 아니라, 프론트엔드 개발자로서 어떤 문제를 중요하게 보고, 어떤 판단으로 해결했는지가 분명하게 보이는 포트폴리오를 완성하는 것입니다.

## 전체 목표

- 메인 페이지는 짧고 강하게 읽히도록 정리합니다.
- 프로젝트 상세 페이지는 문제 해결 과정이 깊게 보이도록 고도화합니다.
- 경력, 역량, 성과 수치는 면접에서 설명 가능한 수준으로 정리합니다.
- 모바일과 데스크톱에서 모두 안정적으로 보이게 QA합니다.
- 최종 결과는 `dev`에 push하고, GitHub Actions를 통해 `master` 배포 브랜치에 자동 반영합니다.

## 작업 원칙

- 작업은 항상 `dev` 브랜치에서 진행합니다.
- `master`에서 `git merge dev`를 하지 않습니다.
- `.next/`와 `out/`은 `dev`에 커밋하지 않습니다.
- 커밋은 변경 목적별로 나눕니다.
- 커밋 메시지 본문은 한국어로 작성합니다.
- 사소한 UI 변경은 문서에 과하게 남기지 않습니다.

## Day 1: 2026-06-25 목요일

예상 작업량: 7h

### 준비: 0.5h

- 현재 사이트를 처음 보는 사람처럼 훑어봅니다.
- 메인 페이지, 프로젝트 목록, 상세 페이지에서 어색한 부분을 메모합니다.
- 지원하고 싶은 포지션 방향을 하나로 정합니다.

산출물:

- 목표 포지션 한 줄
- 강조할 강점 3개
- 수정 우선순위 메모

### 목표 포지션과 첫 화면 메시지 정리: 2h

- 첫 화면 헤드라인을 다시 작성합니다.
- 소개 문장을 2~3개 버전으로 작성합니다.
- 대표 성과 3개가 실제로 설명 가능한지 점검합니다.

확인 질문:

- 5초 안에 어떤 개발자인지 보이는가?
- 사용자 흐름, 제품 안정성, 성능 개선 중 무엇이 가장 먼저 보이는가?
- 추상적인 말보다 실제 강점이 드러나는가?

### 프로젝트 우선순위 재배치: 2h

- 대표 프로젝트 2~3개를 정합니다.
- 회사 프로젝트와 개인 프로젝트의 노출 순서를 정합니다.
- 메인 페이지에 남길 요약과 상세 페이지로 보낼 내용을 분리합니다.

추천 기준:

- 실무 난이도가 보이는 프로젝트
- 직접 판단한 흔적이 보이는 프로젝트
- 프론트엔드 역량과 연결되는 프로젝트
- 면접에서 깊게 설명할 수 있는 프로젝트

### 메인 페이지 수정: 2h

- 첫 화면 문구를 반영합니다.
- 프로젝트 카드 순서와 요약 문구를 조정합니다.
- 불필요하게 긴 설명은 상세 페이지로 이동합니다.

완료 기준:

- 메인 페이지가 한 번에 훑어 읽힙니다.
- 대표 프로젝트가 먼저 눈에 들어옵니다.
- 과장되거나 근거가 약한 표현이 줄어듭니다.

### 정리: 0.5h

- 변경 내용을 확인합니다.
- 필요하면 `npm run build`를 실행합니다.
- 커밋할 단위를 나눕니다.

예상 커밋:

```txt
feat(content): 첫 화면 메시지 정리
feat(portfolio): 프로젝트 노출 순서 조정
```

## Day 2: 2026-06-26 금요일

예상 작업량: 7.5h

### 전날 결과 리뷰: 0.5h

- 메인 페이지를 다시 읽어봅니다.
- 어색한 문장과 과하게 긴 문장을 표시합니다.
- 상세 페이지에서 먼저 고칠 프로젝트를 정합니다.

### 대표 프로젝트 상세 페이지 1차 고도화: 2.5h

대표 프로젝트 1~2개를 아래 구조로 다시 씁니다.

```txt
문제 상황
내가 맡은 역할
고민한 선택지
선택한 이유
실제 구현
결과 또는 배운 점
```

중점:

- "무엇을 적용했다"보다 "왜 그 방법을 선택했는지"를 먼저 씁니다.
- 처음에 막혔던 지점이나 trade-off를 남깁니다.
- 공개할 수 없는 회사 정보는 역할, 제약, 결과 중심으로 표현합니다.

### 수치와 근거 점검: 2h

현재 들어간 수치를 하나씩 검토합니다.

- `3초 -> 1초`
- `설정 오류율 20% 감소`
- `위치 정확도 70% 향상`
- `설치 오류 5건 -> 0건`

확인 질문:

- 어떤 조건에서 측정했는가?
- 면접에서 질문받으면 설명할 수 있는가?
- 근거가 약하면 표현을 완화해야 하는가?

### 상세 페이지 문장 정리: 2h

- 상세 페이지의 문장 톤을 부드럽게 정리합니다.
- 이력서식 나열을 줄이고, 판단 흐름이 보이게 바꿉니다.
- placeholder가 필요한 위치를 표시합니다.

예시:

```txt
[성능최적화 비교 이미지 필요]
[분석 설정 위자드 화면 캡처 필요]
[회원가입 프로세스 동영상 필요]
```

### 정리: 0.5h

- 대표 프로젝트 상세 페이지를 다시 읽습니다.
- 필요하면 `npm run build`를 실행합니다.
- 커밋 단위를 나눕니다.

예상 커밋:

```txt
feat(content): 대표 프로젝트 상세 설명 고도화
docs: 포트폴리오 자료 준비 항목 정리
```

## Day 3: 2026-06-27 토요일

예상 작업량: 7h

### 전체 흐름 점검: 0.5h

- 메인 페이지에서 상세 페이지로 이어지는 흐름을 확인합니다.
- 프로젝트별 설명이 서로 겹치지 않는지 봅니다.
- 부족한 프로젝트와 과하게 긴 프로젝트를 나눕니다.

### 나머지 프로젝트 상세 페이지 정리: 2h

- 개인 프로젝트와 팀 프로젝트 상세 내용을 정리합니다.
- 각 프로젝트마다 한 가지 핵심 문제를 분명히 만듭니다.
- 기술 스택보다 문제 해결 맥락을 먼저 보여줍니다.

완료 기준:

- 각 프로젝트가 서로 다른 강점을 보여줍니다.
- 모든 상세 페이지가 같은 구조로 읽힙니다.
- 프로젝트별 placeholder가 명확합니다.

### 경력 섹션 재구성: 1.5h

- 회사 경험을 단순 소개가 아니라 책임과 문제 유형 중심으로 정리합니다.
- 의료기기 SW의 보안, 추적성, 배포 안정성 경험을 자연스럽게 설명합니다.
- 공개할 수 없는 화면 자료는 문장으로 맥락을 보완합니다.

중점:

- 어떤 환경에서 일했는지
- 어떤 책임을 맡았는지
- 어떤 품질 기준을 고려했는지
- 사용자와 팀에 어떤 변화가 있었는지

### 역량 섹션 재구성: 2h

기술명을 나열하기보다 문제 해결 능력 기준으로 묶습니다.

현재 기준:

- Frontend Architecture
- UX Engineering
- Performance
- Delivery

개선 방향:

- 각 역량이 어떤 프로젝트와 연결되는지 표시합니다.
- 기술 스택은 보조 정보로 둡니다.
- 실제 사례 없이 추상적인 문장은 줄입니다.

### 정리: 1h

- 경력과 역량 섹션을 소리 내어 읽어봅니다.
- 너무 딱딱한 문장을 표시합니다.
- 다음 날 레이아웃 수정 목록을 만듭니다.

예상 커밋:

```txt
feat(content): 경력과 역량 설명 재구성
feat(project): 프로젝트 상세 콘텐츠 보강
```

## Day 4: 2026-06-28 일요일

예상 작업량: 6.5h

### 디자인 점검: 0.5h

- 데스크톱 첫 화면을 확인합니다.
- 프로젝트 카드 간격과 문단 길이를 확인합니다.
- 색상, 폰트 크기, 버튼 크기가 과하지 않은지 봅니다.

### 메인 페이지 레이아웃 밀도 조정: 2h

- 첫 화면의 문장 길이와 여백을 조정합니다.
- 프로젝트 카드가 배너처럼 읽히도록 정리합니다.
- 섹션 제목과 설명의 계층을 더 명확하게 만듭니다.

완료 기준:

- 첫 화면에서 핵심 정보가 잘 보입니다.
- 프로젝트 목록이 답답하지 않습니다.
- CTA와 연락처가 자연스럽게 보입니다.

### 상세 페이지 레이아웃 조정: 2h

- 상세 페이지의 문단 폭과 여백을 조정합니다.
- placeholder 영역이 임시 문구처럼 보이되, 너무 거슬리지 않게 만듭니다.
- 모바일에서도 제목과 문단이 겹치지 않는지 확인합니다.

### 반응형 QA: 1.5h

확인할 화면:

- 데스크톱 첫 화면
- 데스크톱 프로젝트 목록
- 데스크톱 프로젝트 상세 페이지
- 모바일 첫 화면
- 모바일 프로젝트 목록
- 모바일 프로젝트 상세 페이지

확인 항목:

- 가로 스크롤이 생기지 않는가?
- 버튼과 카드 텍스트가 넘치지 않는가?
- 프로젝트 상세 페이지로 이동이 자연스러운가?
- 폰트가 Pretendard로 잘 적용되는가?

### 정리: 0.5h

- `npm run build`를 실행합니다.
- 레이아웃 변경만 별도 커밋으로 나눕니다.

예상 커밋:

```txt
style(portfolio): 메인 페이지 레이아웃 밀도 조정
style(project): 상세 페이지 반응형 스타일 개선
```

## Day 5: 2026-06-29 월요일

예상 작업량: 5.5h

### 전체 문장 검수: 1h

- 모든 페이지를 처음부터 끝까지 읽습니다.
- 어색한 문장, 반복되는 표현, 과장된 표현을 줄입니다.
- 맞춤법과 띄어쓰기를 확인합니다.

점검 기준:

- 문장이 너무 이력서처럼 딱딱하지 않은가?
- "저는 ~했습니다"가 반복되지 않는가?
- 판단과 이유가 보이는가?
- 성과가 설명 가능한 수준인가?

### 채용자 관점 점검: 1.5h

처음 보는 사람이 아래 질문에 답할 수 있는지 확인합니다.

- 어떤 개발자인가?
- 어떤 문제를 잘 푸는가?
- 실무에서 어떤 역할을 맡았는가?
- 우리 팀에 들어오면 어떤 기여를 할 수 있는가?
- 프로젝트를 실제로 깊게 이해하고 있는가?

### 최종 QA: 1.5h

아래 항목을 확인합니다.

- `npm run build` 성공
- 메인 페이지 링크 동작
- 프로젝트 상세 페이지 진입
- 외부 링크 동작
- 데스크톱 레이아웃
- 모바일 레이아웃
- GitHub Pages 배포 흐름
- README, AGENTS, SKILL, PORTFOLIO_PLAN 문서 최신성

### 커밋 정리: 1h

변경사항을 목적별로 나눠 커밋합니다.

예상 커밋:

```txt
fix(content): 포트폴리오 문장 표현 정리
test: 최종 빌드 검증
docs: 포트폴리오 개선 계획 갱신
```

### 배포 확인: 0.5h

- `dev`에 push합니다.
- GitHub Actions의 `Deploy portfolio` 실행 여부를 확인합니다.
- `pages-build-deployment`가 이어서 실행되는지 확인합니다.
- 실제 사이트 `https://dmswl6310.github.io`에서 반영 상태를 확인합니다.

완료 기준:

- 실제 사이트에서 README가 아니라 포트폴리오 페이지가 보입니다.
- 메인 페이지와 상세 페이지가 정상 동작합니다.
- 모바일에서 읽기 불편한 부분이 없습니다.

## 최종 산출물

2026년 6월 29일 작업 종료 시점에는 아래 상태를 목표로 합니다.

- 첫 화면 메시지가 명확합니다.
- 대표 프로젝트의 우선순위가 분명합니다.
- 상세 페이지에 문제 해결 과정이 드러납니다.
- 경력과 역량 섹션이 프로젝트와 연결됩니다.
- 필요한 이미지와 영상 placeholder가 정리되어 있습니다.
- 데스크톱과 모바일 QA가 완료되어 있습니다.
- `dev` 브랜치에 최종 커밋이 push되어 있습니다.
- GitHub Pages 실제 사이트가 정상 반영되어 있습니다.
