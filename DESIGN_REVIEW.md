# Portfolio Design Review

이 문서는 2026년 6월 24일 기준으로 포트폴리오 사이트의 디자인과 레이아웃을 점검하고, 바로 수정하면 좋을 항목을 정리한 문서입니다.

`PORTFOLIO_PLAN.md`는 2026년 6월 25일부터 6월 29일까지의 콘텐츠 개선 커리큘럼으로 유지하고, 이 문서는 그 전에 정리할 디자인 방향만 다룹니다.

### 관찰한 좋은 패턴

- 첫 화면에서 이름, 직무, 방향성이 바로 보입니다.
- 프로젝트 목록은 제목, 한 줄 요약, 링크, 기술 스택 중심으로 빠르게 훑게 합니다.
- 경력과 프로젝트는 단순 나열보다 시간, 역할, 결과가 함께 보일 때 신뢰감이 생깁니다.
- 좋은 포트폴리오는 모든 것을 화려하게 만들기보다, 핵심 콘텐츠가 잘 읽히도록 여백과 계층을 정리합니다.
- 시각 자료가 있는 프로젝트는 기억에 더 잘 남지만, 자료가 없을 때는 문제 상황과 판단 과정이 그 역할을 대신해야 합니다.
- 합격 포트폴리오 모음형 글은 디자인 자체보다 채용자가 빠르게 비교할 수 있는 정보 구조를 참고하는 용도로 봅니다.

## Current Site Review

### 장점

- Next.js SSG 구조와 프로젝트 상세 페이지 분리가 이미 되어 있습니다.
- 메인 페이지에서 프로젝트를 배너형으로 보여주는 방향은 유지할 가치가 있습니다.
- Pretendard 적용으로 한글 가독성의 기본은 괜찮습니다.
- 회사 프로젝트와 개인 프로젝트가 모두 있어 실무성과 주도성을 함께 보여줄 수 있습니다.
- 성과 수치가 있어 채용자가 빠르게 훑을 때 눈에 걸리는 지점이 있습니다.

### 아쉬운 점

- 첫 화면이 아직 시각적으로 강하게 기억되지는 않습니다.
- 대표 프로젝트와 보조 프로젝트의 무게 차이가 크지 않아 우선순위가 약하게 느껴질 수 있습니다.
- 성과 카드가 숫자는 강하지만, 어떤 프로젝트와 연결되는지 한 번 더 추적해야 합니다.
- 상세 페이지의 placeholder 영역은 필요하지만, 현재는 임시 박스 느낌이 강할 수 있습니다.
- 경력과 역량 섹션이 프로젝트와 직접 연결되지 않아 설득력이 분산될 수 있습니다.
- 전체 톤이 안정적이지만, 개인의 문제 해결 방식이 디자인적으로 드러나는 장치가 더 있으면 좋습니다.

## Design Improvement Direction

### 1. 첫 화면에 기억점을 만듭니다

현재 첫 화면은 텍스트와 성과 카드 중심입니다.
여기에 개발자로서의 방향을 압축해서 보여주는 시각 장치를 추가합니다.

개선 아이디어:

- `문제 발견 -> 판단 -> 구현 -> 검증` 흐름을 작은 process strip으로 보여줍니다.
- 대표 성과 카드를 프로젝트명과 연결합니다.
- 첫 화면 오른쪽 카드 영역을 단순 숫자 카드에서 작업 방식 보드처럼 재구성합니다.

목표:

- 방문자가 첫 화면만 보고도 사용자 흐름과 안정성을 중요하게 보는 프론트엔드 개발자라고 이해하게 합니다.

### 2. 프로젝트 목록에 우선순위를 만듭니다

현재 프로젝트 카드는 모두 같은 형태로 나열됩니다.
대표 프로젝트 2~3개와 나머지 프로젝트를 시각적으로 구분합니다.

개선 아이디어:

- 상단에 `Featured Projects` 영역을 둡니다.
- 대표 프로젝트는 큰 배너 카드로 보여주고, 나머지는 compact list로 정리합니다.
- 회사 프로젝트, 개인 프로젝트, 팀 프로젝트를 badge로 구분합니다.
- 각 프로젝트 카드에 문제 유형을 표시합니다.

예시 문제 유형:

```txt
UX Flow
Performance
Security UX
Location Accuracy
Concurrency
Delivery
```

목표:

- 채용자가 먼저 봐야 할 프로젝트를 고민 없이 따라가게 합니다.

### 3. 성과 수치와 프로젝트를 연결합니다

성과 숫자는 강하지만, 단독으로 보면 맥락이 약할 수 있습니다.

개선 아이디어:

- 성과 카드에 관련 프로젝트 링크를 붙입니다.
- 숫자 아래에 측정 기준을 한 줄로 추가합니다.
- 근거가 약한 수치는 `기여`, `약`, `내부 기준`처럼 표현을 조정합니다.

목표:

- 면접에서 질문받아도 설명 가능한 수치만 남깁니다.

### 4. 상세 페이지를 case study처럼 다듬습니다

상세 페이지는 포트폴리오의 핵심입니다.
현재 구조는 좋지만, 더 읽기 쉬운 case study 형식으로 다듬을 수 있습니다.

개선 아이디어:

- 상세 페이지 상단에 `역할`, `기간`, `기여도`, `핵심 성과`, `기술` 요약 박스를 추가합니다.
- 본문은 `문제`, `판단`, `구현`, `결과` 순서로 유지합니다.
- 긴 리스트는 timeline 또는 step 형태로 바꿉니다.
- placeholder 영역은 단순 박스가 아니라 나중에 들어갈 자료의 목적이 보이게 씁니다.

예시:

```txt
[분석 설정 위자드 화면 캡처 필요]
목적: 8개 설정 항목이 3단계로 분리된 흐름을 보여주기
```

목표:

- 단순 작업 목록이 아니라 실제로 깊이 고민한 프로젝트처럼 읽히게 합니다.

### 5. 경력과 역량을 프로젝트와 연결합니다

역량 섹션이 독립적으로 있으면 흔한 기술 소개처럼 보일 수 있습니다.

개선 아이디어:

- 각 역량 카드 하단에 관련 프로젝트 링크를 표시합니다.
- 경력 섹션에는 회사에서 맡은 책임을 3개 키워드로 요약합니다.
- `의료기기 SW`, `레거시 개선`, `신규 웹앱`, `배포 안정화`처럼 경험의 결을 드러냅니다.

목표:

- 기술 스택보다 실제로 어떤 상황에서 썼는지가 보이게 합니다.

### 6. 시각 톤은 심플하게 유지하되 밀도를 조정합니다

현재 색상은 안정적이지만 조금 평평하게 느껴질 수 있습니다.

개선 아이디어:

- 배경은 유지하되 섹션마다 아주 약한 대비를 줍니다.
- 카드 내부의 제목, 메타, 성과, 기술 스택 계층을 더 선명하게 나눕니다.
- hover 효과는 과하지 않게 유지하되, 프로젝트 카드의 클릭 가능성은 더 분명히 합니다.
- 모바일에서는 카드 간격과 문단 길이를 더 줄입니다.

목표:

- 심플하지만 기억에 남는 포트폴리오로 만듭니다.

## 2026-06-24 Design Revision Priority

오늘 디자인과 레이아웃을 고친다면 아래 순서로 진행합니다.

```txt
1. 첫 화면 hero 구조 재정리
2. 대표 프로젝트와 보조 프로젝트 구분
3. 성과 카드와 프로젝트 연결
4. 프로젝트 상세 페이지 case study 구조 보강
5. 경력/역량 섹션과 프로젝트 연결
6. placeholder 디자인 정리
7. 모바일 레이아웃 QA
```

가장 먼저 고칠 항목은 첫 화면과 프로젝트 목록입니다.
첫 화면과 프로젝트 우선순위가 정해져야 나머지 스타일의 기준이 생깁니다.

## 합격 포트폴리오 관점 체크리스트

- 첫 화면에서 이름, 직무, 강점이 즉시 보이는가?
- 프로젝트가 많이 했다는 인상보다 깊게 고민했다는 인상으로 읽히는가?
- 대표 프로젝트와 보조 프로젝트의 우선순위가 분명한가?
- 각 프로젝트에 역할, 문제, 해결 과정, 결과가 빠지지 않는가?
- 기술 스택이 단순 나열이 아니라 실제 문제 해결과 연결되는가?
- 링크, GitHub, 배포 사이트, 자료 placeholder의 목적이 명확한가?
- 모바일에서도 채용자가 빠르게 훑을 수 있는가?
- 공개할 수 없는 회사 프로젝트도 책임 범위와 판단 과정이 충분히 보이는가?
