# Portfolio Site Skill

이 문서는 포트폴리오 사이트를 반복적으로 수정할 때 참고하는 작업 절차입니다.

## 목적

황은지 포트폴리오 사이트를 Next.js, TypeScript, SSG 구조 안에서 일관되게 유지합니다.

## 사용 상황

다음 작업에서는 이 문서를 참고합니다.

- 포트폴리오 문구 수정
- 프로젝트 추가 또는 삭제
- 프로젝트 상세 페이지 구성 변경
- 메인 페이지 배너 구성 변경
- GitHub Pages 배포 설정 변경
- 커밋 단위 정리

## 작업 절차

1. 현재 브랜치와 작업 상태를 확인합니다.

```bash
git status --short --branch
```

2. 콘텐츠 변경이면 `data/portfolio.ts`를 먼저 수정합니다.

3. 화면 구조 변경이면 아래 파일을 확인합니다.

```txt
app/page.tsx
app/projects/[slug]/page.tsx
app/layout.tsx
app/globals.css
```

4. 프로젝트 상세 페이지가 추가되는 경우 `projects` 배열에 고유한 `slug`를 추가합니다.

5. 라우팅, 빌드 설정, 데이터 구조를 바꿨다면 빌드를 확인합니다.

```bash
npm run build
```

6. 빌드 후 생성되는 `.next/`와 `out/`은 `dev`에 커밋하지 않습니다.

7. 커밋은 변경 성격에 따라 나눕니다.

## 콘텐츠 작성 기준

프로젝트 설명은 단순히 무엇을 적용했는지보다 아래 흐름을 우선합니다.

```txt
문제의 배경
왜 그 문제가 중요했는지
어떤 선택지를 고민했는지
무엇을 실행했는지
결과가 어떻게 달라졌는지
```

문장은 이력서보다 부드럽게 작성하되, 성과와 판단 근거는 구체적으로 유지합니다.

## 프로젝트 카드 기준

메인 페이지의 프로젝트 카드는 긴 설명을 넣지 않습니다.

포함할 내용:

- 프로젝트 타입
- 제목
- 한두 문장의 요약
- 대표 성과
- 기술 스택

자세한 내용은 상세 페이지에 둡니다.

## 상세 페이지 기준

상세 페이지는 아래 섹션을 유지합니다.

- 문제와 배경
- 제가 중요하게 본 지점
- 실행한 일
- 결과

이미지나 영상이 아직 없을 때는 placeholder 문구를 유지합니다.

예시:

```txt
[성능최적화 이미지 필요]
[회원가입 프로세스 동영상 필요]
```

## 배포 기준

소스 브랜치는 `dev`입니다.

`dev`에 push하면 GitHub Actions가 정적 사이트를 빌드하고,
생성된 `out/` 결과물만 `master` 브랜치에 반영합니다.

GitHub Pages는 `master` 브랜치 루트를 서비스합니다.
따라서 `master`는 배포 결과물 전용 브랜치로 보고 직접 소스 작업을 하지 않습니다.

## 커밋 기준

커밋 헤더는 conventional commit 형식을 따릅니다.

본문은 한국어로 작성합니다.

예시:

```txt
feat(content): 프로젝트 데이터 추가

포트폴리오 메인과 상세 페이지에서 사용할 프로젝트 정보를 추가한다.
긴 설명은 상세 페이지에서만 보이도록 데이터 구조를 나눈다.
```

커밋은 가능한 한 다음 단위로 나눕니다.

- 설정
- 데이터
- 메인 페이지
- 상세 페이지
- 스타일
- 배포
- 문서
